### Frontend ↔ Backend Communication

The frontend communicates with the backend exclusively through **tRPC** over HTTP batch requests:

- **Client:** `client/src/main.tsx` creates a tRPC client using `httpBatchLink` pointing to `/api/trpc` with `credentials: "include"` for cookie-based auth.
- **Server:** `server/_core/index.ts` mounts the tRPC handler at `/api/trpc` using Express middleware via `@trpc/server/adapters/express`.
- **Auth:** Every request goes through `createContext()` in `server/_core/context.ts`, which calls `sdk.authenticateRequest()` to extract the JWT from the `app_session_id` cookie. If valid, `ctx.user` is populated; otherwise it's `null`.
- **Procedures:** Three levels defined in `server/_core/trpc.ts`:
  - `publicProcedure` — no auth required
  - `protectedProcedure` — requires valid session (any role)
  - `adminProcedure` — requires valid session + `role === "admin"`

### Database

- **Engine:** MySQL 8 on Railway (private networking via `mysql.railway.internal:3306`)
- **ORM:** Drizzle ORM with `drizzle-orm/mysql2`
- **Schema:** `drizzle/schema.ts` defines all tables (users, brands, posts, social_accounts, notifications, services, events, chatbot_flows, chatbot_leads, error_logs, invites, onboarding_state)
- **Migrations:** Generated by `drizzle-kit generate`, applied at **server startup** via `server/services/migrate.ts` (NOT during build — see Common Issues)
- **Connection:** `server/db.ts` creates a singleton connection using `DATABASE_URL`. Includes retry logic — resets on failure so subsequent calls retry.

### Railway Deployment

Railway auto-deploys from the `main` branch on GitHub. The configuration is in `railway.toml`:

```toml
[build]
builder = "nixpacks"
buildCommand = "pnpm install && pnpm build"

[deploy]
startCommand = "NODE_ENV=production node dist/index.js"
healthcheckPath = "/api/health"
healthcheckTimeout = 60
restartPolicyType = "on_failure"
restartPolicyMaxRetries = 3
```

**Build step:** `pnpm build` runs two commands chained in `package.json`:
1. `vite build` — bundles the React frontend into `dist/public/`
2. `esbuild server/_core/index.ts` — bundles the server into `dist/index.js`

**Start step:** `NODE_ENV=production node dist/index.js` — Express serves the static frontend from `dist/public/` and handles API requests.

## 2. Key Logic & Workflows

### Authentication Flow

Standalone email/password auth (Manus OAuth has been fully removed):

1. **Register** (`POST /api/auth/register`): Email + password + name → bcrypt hash (12 rounds) → creates user with `openId = local_{uuid}` → signs JWT → sets `app_session_id` cookie.
2. **Login** (`POST /api/auth/login`): Email + password → bcrypt compare → signs JWT → sets cookie.
3. **Session:** JWT signed with `JWT_SECRET` using HS256 via `jose` library. Cookie is `app_session_id`, expires in 1 year.
4. **Password Reset** (`POST /api/auth/reset-password-request` + `POST /api/auth/reset-password`): Generates a crypto token, stores it on the user record with expiry.
5. **Logout** (`POST /api/auth/logout`): Clears the session cookie.

### Admin vs Client Roles

- **Admin:** Full access — manage all brands, approve onboarding, access AI Engine, view all posts/users, system health monitoring.
- **Client (managed/premium):** Can only see their own brand's posts, calendar, events, notifications. Cannot access other brands or admin tools.
- Role is stored in `users.role` column. The `adminProcedure` middleware enforces admin-only access on the server side.

### User Onboarding & Brand Setup

1. New user registers at `/signup` → redirected to `/onboarding`.
2. Onboarding is a multi-step wizard (`client/src/pages/Onboarding.tsx`) that collects:
   - Business name, industry, location, website
   - Brand voice settings (tone, style, keywords, avoid words, sample posts, custom instructions)
   - Social media account connections
3. State is saved incrementally via `onboarding.saveStep` tRPC mutation.
4. On completion, `onboarding.complete` creates the brand record with `status: "pending"`.
5. Admin is notified (via Forge API or SMTP fallback or console log).
6. Admin approves at `/admin/onboarding-approval` via `onboarding.approve` → sets `brand.isActive = true`.
7. Client can then access their dashboard.

### Brand Voice Learning & Content Generation

The brand voice system works through the `voiceSettings` JSON field on the `brands` table:

```typescript
voiceSettings: {
  tone: string;        // e.g., "Professional but approachable"
  style: string;       // e.g., "Direct, value-driven"
  keywords: string[];  // Terms to use frequently
  avoidWords: string[]; // Terms to never use
  samplePosts: string[]; // Example posts for voice reference
  customInstructions: string; // Freeform guidance
}
```

**Content generation flow** (`server/services/contentEngine.ts`):

1. `generatePost()` is called with brand name, content type, and voice settings.
2. A **content type** is selected — either explicitly requested or auto-rotated from a weighted pool:
   - General types: `hey_tony`, `hook_solve`, `auditor_showcase`, `local_tips`, `machine_series`, `print_digital`
   - Shopify types: `shopify_product` (if Shopify connected)
   - Service types: `service_spotlight` (if services configured)
3. A system prompt is built combining the brand voice settings, format-specific instructions, and source context (Shopify product data or service data).
4. The prompt is sent to the LLM (`server/_core/llm.ts`) using `invokeLLM()` with a JSON schema response format.
5. The LLM returns `{ postContent, imagePrompt }` — the post text and a suggested image generation prompt.
6. The response is parsed and returned to the frontend.

**Batch generation** (`fillSchedule` procedure in `routers.ts`):
- Takes a date range and posting frequency.
- Generates multiple posts with staggered `scheduledAt` times.
- Each post gets AI-generated content + optional AI-generated image.
- Posts are created with status `"scheduled"`.

### Image Generation Workflow

**Single image** (`generateImage` in `server/_core/imageGeneration.ts`):
1. Sends prompt to Manus Forge ImageService API (`BUILT_IN_FORGE_API_URL/images.v1.ImageService/GenerateImage`).
2. Receives base64-encoded image.
3. Uploads to Forge Storage proxy (`storagePut()` in `server/storage.ts`).
4. Returns the public URL.

**Smart image** (`generateSmartImage` procedure):
- Enhances the prompt with brand-specific styling (dark navy/tech blue color scheme).
- Uses the same pipeline as above.

**Carousel images:**
- `generateCarousel` procedure generates multi-slide content via LLM.
- Each slide gets its own image prompt.
- Images are generated individually and assembled into a carousel post.

> **Important:** Both image generation and storage currently depend on `BUILT_IN_FORGE_API_URL` and `BUILT_IN_FORGE_API_KEY` (Manus platform APIs). These need to be replaced with direct OpenAI/DALL-E or another image API + S3 storage for fully independent operation.

### Content Scheduling & Auto-Posting

**Cron engine** (`server/services/cronEngine.ts`):

1. An in-process scheduler runs every 5 minutes (configurable, disabled with `DISABLE_IN_PROCESS_CRON=true`).
2. It queries for posts with status `"scheduled"` or `"approved"` where `scheduledAt <= now`.
3. For each due post:
   - Looks up the brand's connected social accounts.
   - Publishes to Facebook via `publishToFacebook()` and/or Instagram via `publishToInstagram()`.
   - Supports single-image posts and carousel posts.
   - On success: updates post status to `"published"`.
   - On failure: increments `retryCount`, logs error, retries up to 3 times with exponential backoff.
   - Detects token expiration errors and creates notifications for the admin.
4. External cron trigger: `POST /api/cron/publish` (requires `CRON_SECRET` header for auth).

### Stripe Integration

**Checkout flow** (`server/services/stripe.ts`):
1. `POST /api/stripe/create-checkout-session` — creates a Stripe Checkout session for managed or premium tier.
2. Uses `STRIPE_PRICE_MANAGED` and `STRIPE_PRICE_PREMIUM` environment variables for price IDs.
3. Creates/finds Stripe customer, stores `stripeCustomerId` on user record.

**Webhook handling** (`POST /api/stripe/webhook`):
- Processes `checkout.session.completed`, `customer.subscription.updated`, `customer.subscription.deleted`, `invoice.payment_failed`.
- Updates user tier and subscription status in the database.
- Requires `STRIPE_WEBHOOK_SECRET` for signature verification.

**Customer portal:** `POST /api/stripe/create-portal-session` — redirects to Stripe's self-service portal.

**Graceful degradation:** All Stripe features check `isStripeConfigured()` first. If `STRIPE_SECRET_KEY` is not set, Stripe features are silently disabled.

### LLM Integration

The LLM is accessed through `server/_core/llm.ts`:

- **Model:** `gemini-2.5-flash` (hardcoded in `invokeLLM()`)
- **API:** OpenAI-compatible chat completions endpoint
- **URL resolution:** Uses `BUILT_IN_FORGE_API_URL/v1/chat/completions` if set, otherwise falls back to `https://forge.manus.im/v1/chat/completions`
- **Auth:** Bearer token using `BUILT_IN_FORGE_API_KEY`
- **Features:** Supports tool calling, JSON schema response format, vision (image URLs), and file content

> **Migration note:** To use a different LLM provider, modify `resolveApiUrl()` and `assertApiKey()` in `server/_core/llm.ts`. The interface is OpenAI-compatible, so any provider with that API format will work.

```bash
# Build frontend (Vite) + backend (esbuild)
pnpm build

# Start production server
NODE_ENV=production node dist/index.js
```

### Deploying to Railway

Railway auto-deploys when you push to the `main` branch:

```bash
# Stage and commit changes
git add -A
git commit -m "Descriptive commit message"

# Push to GitHub (triggers Railway auto-deploy)
git push origin main
```

**What happens on Railway:**
1. Railway detects the push via GitHub webhook.
2. Nixpacks builds the Docker image using `railway.toml` config.
3. Build runs: `pnpm install && pnpm build` (Vite + esbuild).
4. Container starts: `NODE_ENV=production node dist/index.js`.
5. Server runs Drizzle migrations at startup (`server/services/migrate.ts`).
6. Railway hits `/api/health` — if it returns 200 within 60s, deploy succeeds.
7. Old container is replaced with the new one (zero-downtime).

### Checking Deploy Logs

1. Go to Railway dashboard → the-signal project → the-signal service.
2. Click on the latest deployment.
3. **Build Logs** tab: Shows Nixpacks build output (install, compile).
4. **Deploy Logs** tab: Shows runtime server output (startup, migrations, errors).

### Manual Redeploy

If auto-deploy doesn't trigger, push an empty commit:

```bash
git commit --allow-empty -m "Trigger redeploy"
git push origin main
```

## 5. Common Issues & Fixes

### DATABASE_URL Not Available During Build

**Problem:** `drizzle-kit generate && drizzle-kit migrate` (the `db:push` script) requires `DATABASE_URL`, but Railway environment variables are only available at **runtime**, not during the Docker build phase. The internal hostname `mysql.railway.internal` only resolves inside Railway's private network.

**Solution:** Database migrations run at **server startup** via `server/services/migrate.ts`, not during build. The `railway.toml` `buildCommand` must NOT include `pnpm db:push`.

### Dev Dependencies Pruned by Nixpacks

**Problem:** Nixpacks prunes `devDependencies` after the build step. If the server bundle (`dist/index.js`) has top-level static imports for dev-only packages (Vite, Tailwind plugins, etc.), the server crashes on startup.

**Solution:** Dev-only imports in `server/_core/vite.ts` use dynamic `import()` with string concatenation to prevent esbuild from bundling them. These are only loaded in development mode.

### Healthcheck Failure

**Problem:** Railway's healthcheck hits `/api/health` and expects a 200 response within 60 seconds.

**Common causes:**
- Server binding to `localhost` instead of `0.0.0.0` (must bind to all interfaces).
- Startup crash from missing env vars or import errors.
- Database connection timeout blocking server startup.

**Solution:** The healthcheck endpoint is registered early in the Express middleware chain. Database connection is non-blocking — the server starts even if the DB is unavailable.

### Node.js Version

**Requirement:** Node.js >= 20.0.0 (specified in `package.json` engines and `.node-version`).

**Note:** Vite v7 requires Node.js 20.19+ or 22.12+. Railway currently provides Node 22.11.0 via Nixpacks, which triggers a Vite warning but still works.

### Image Generation & Storage Dependency on Manus

**Problem:** Both `imageGeneration.ts` and `storage.ts` depend on `BUILT_IN_FORGE_API_URL` and `BUILT_IN_FORGE_API_KEY` (Manus platform APIs). Without these, image generation and file storage will fail.

**Migration path:** Replace `generateImage()` with direct calls to OpenAI DALL-E or another image API. Replace `storagePut()`/`storageGet()` with direct AWS S3 SDK calls using your own bucket.

### LLM Dependency on Manus Forge

**Problem:** `llm.ts` sends requests to the Manus Forge proxy (`BUILT_IN_FORGE_API_URL/v1/chat/completions`). The model is hardcoded as `gemini-2.5-flash`.

**Migration path:** Set `BUILT_IN_FORGE_API_URL` to `https://api.openai.com` and `BUILT_IN_FORGE_API_KEY` to your OpenAI API key. Change the model in `llm.ts` line ~270 from `gemini-2.5-flash` to your preferred model (e.g., `gpt-4.1-mini`). The API format is already OpenAI-compatible.
